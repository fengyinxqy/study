# Redis 学习笔记

> 学习者背景：2 年 Vue 前端经验，Redis 零基础  
> 学习方式：苏格拉底式提问法 —— 通过思考回答问题来建立理解

---

## 一、Redis 的使用场景

### 问 1：分布式系统中如何解决 session 共享问题？

**问题**：如果后端服务部署在多台服务器上（比如 10 台），用户第一次请求被分配到服务器 A，session 存在 A 上；但第二次请求被分配到服务器 B，这时候会发生什么问题？

**答案**：服务器 B 无法获取服务器 A 上的 session 数据，用户需要重新登录。

**解决方案**：使用一个"公共存储区域"，所有服务器都能访问 —— 这就是 Redis 的核心使用场景之一。

```
用户登录 → Session 存入 Redis → 任意服务器都能从 Redis 读取
```

---

## 二、数据过期策略

### 问 2：Session 数据有什么特点？需要永久保存吗？

**答案**：Session 数据有**时效性**，过期后应该被清除。

类比前端：类似 cookie 的过期机制，可以设置过期时间。

---

### 问 3：Redis 如何实现"自动过期"？

**问题**：如果某个 session 设置 2 小时过期，2 小时后这个数据是立刻被删除，还是可能还留在内存里一段时间？

**答案**：Redis 使用**组合策略**：

| 策略 | 实现方式 | 优点 | 缺点 |
|------|----------|------|------|
| **惰性删除** | 访问时检查是否过期 | 节省 CPU | 过期数据可能残留 |
| **定期删除** | 每隔一段时间随机抽查删除 | 保证数据最终被清理 | 需要消耗 CPU |

**结论**：单纯用惰性删除会导致内存被占满；单纯用定期删除会消耗大量 CPU。Redis 结合两者优点。

---

## 三、高性能的原因

### 问 4：Redis 为什么比硬盘数据库快？

**问题**：从数据存储位置的角度思考，内存和硬盘，哪个读写更快？

**答案**：**纯内存存储**是 Redis 高性能的核心原因。

```
内存读写：约 100ns 级别
硬盘读写：约 10ms 级别（相差约 10 万倍）
```

---

## 四、持久化策略

### 问 5：Redis 断电后数据会怎样？

**问题**：Redis 数据都存在内存里，如果服务器突然断电，数据会怎么样？

**答案**：内存是**易失的**，断电后数据会丢失。

---

### 问 6：如何保证数据不丢失？

**问题**：如果公司要求 Redis 重启后数据不能丢，该怎么设计？

**方案对比**：

| 方案 | 实现方式 | 优点 | 缺点 |
|------|----------|------|------|
| **全量快照** | 每隔一段时间把内存数据打包存到硬盘 | 简单，恢复快 | 可能丢失 snapshot 间隔内的数据 |
| **每笔记录** | 每次写操作都同步写入硬盘 | 数据不丢失 | 硬盘 IO 成为瓶颈，性能极差 |
| **缓冲区 + 刷盘** | 写操作先记到内存缓冲区，每秒统一刷新到硬盘 | 性能和安全性平衡 | 最多丢失 1 秒数据 |

---

### 问 7：如何权衡性能和安全性？

**场景**：如果同时有 10 万个写请求，每次写操作都同步写入硬盘，硬盘会怎么样？

**答案**：硬盘 IO 会成为瓶颈，导致系统无法处理。

**折中方案**：
- 每次写操作先记到**内存缓冲区**（很快）
- 缓冲区**每秒刷新一次**到硬盘

**结论**：最多丢失 1 秒数据，但性能从 N 次 IO 降到每秒 1 次。

## 六、Redis 数据结构

Redis 有 5 种核心数据结构，每种都对应前端的某种概念。

---

### 问 1：String（简单值存储）

**问题**：你在 Vue 中存储一个简单的用户名（比如 `"zhangsan"`），会怎么定义这个变量？

```js
// 前端
const username = "zhangsan"
```

**答案**：Redis 用 **String** 结构存储：

```bash
SET username "zhangsan"
GET username
```

**典型场景**：
- 缓存简单值（用户名、计数器）
- Session 存储
- 分布式锁（`SETNX`）

---

### 问 2：Hash（对象存储）

**问题**：前端存储一个对象：
```js
const user = {
  name: "zhangsan",
  email: "zhang@example.com",
  age: 25
}
```

Redis 用什么数据结构？

**答案**：使用 **Hash** 结构（类似 JS 的 Object）：

```bash
# 设置字段
HSET user:1001 name "zhangsan"
HSET user:1001 email "zhang@example.com"
HSET user:1001 age 25

# 获取单个字段（类似 user.name）
HGET user:1001 name

# 获取所有字段（类似 Object.entries(user)）
HGETALL user:1001
```

**典型场景**：
- 存储用户信息
- 存储商品详情
- 任何"对象"类型的数据

**注意**：Redis Hash 是**逐个字段存储**，不是存 JSON 字符串。

---

### 问 3：List（队列场景）

**问题**：前端队列操作：
```js
queue.push("msg")    // 从尾部添加
queue.shift()        // 从头部取出（先进先出）
```

Redis List 如何实现？

**答案**：

```bash
# 从尾部插入
RPUSH queue "msg1" "msg2" "msg3"

# 从头部取出（先进先出）
LPOP queue
```

**典型场景**：
- 消息队列（先进先出）
- 任务队列（文件上传/处理）
- 最近浏览记录（后进先出，类似栈）

**命令规律**：
- `L` 开头 = Left（头部）
- `R` 开头 = Right（尾部）

---

### 问 4：Set（去重场景）

**问题**：前端去重：
```js
const tags = new Set(["js", "vue", "redis"])
tags.add("js")  // 不会重复添加
```

Redis Set 的特点是什么？

**答案**：**自动去重**

```bash
# 添加元素（重复添加会自动忽略）
SADD tags "js" "vue" "redis"

# 集合运算
SINTER set1 set2    # 交集（共同好友）
SUNION set1 set2    # 并集（合并标签）
SDIFF set1 set2     # 差集（互粉检测）
```

**典型场景**：
- 标签系统（自动去重）
- 共同好友/共同关注（交集运算）
- 抽奖（随机抽取不重复用户）

---

### 问 5：ZSet（排行榜场景）

**问题**：游戏排行榜需要按分数排序，传统做法每次查询都要 `ORDER BY`。Redis 如何优化？

**答案**：使用 **ZSet**（Sorted Set）—— 自动保持有序

```bash
# 添加元素（带分数）
ZADD leaderboard 100 "player1"
ZADD leaderboard 300 "player2"
ZADD leaderboard 200 "player3"

# 直接获取前 3 名（已排序，无需额外排序）
ZREVRANGE leaderboard 0 2 WITHSCORES
```

**典型场景**：
- 游戏排行榜
- 热门文章（按热度排序）
- 优先级队列

**性能优势**：
- 插入时已排序，查询无需 `ORDER BY`
- 时间复杂度 O(log N)

## 九、持久化深入（RDB vs AOF）

---

### 问 1：RDB 快照（Redis Database）

**问题**：Redis 每 5 分钟保存一次完整快照。如果第 5 分钟刚保存完，第 5 分 30 秒宕机，会丢失多少数据？

**答案**：丢失**第 5 分钟到第 5 分 30 秒**之间的数据。

**RDB 特点**：
| 优点 | 缺点 |
|------|------|
| 保存完整快照，恢复快 | 可能丢失最后一次快照后的数据 |
| 文件紧凑（二进制） | 数据量大时 fork() 慢 |
| 适合备份 | 不支持实时持久化 |

**配置示例**：
```conf
save 300 10000    # 5 分钟内至少 1 万次修改才快照
save 900 1        # 15 分钟内至少 1 次修改才快照
```

---

### 问 2：AOF 持久化（Append Only File）

**问题**：AOF 记录每次写操作，最多会丢失多少数据？文件会持续增长吗？

**答案**：
- 最多丢失**一条命令**（如果配置为每条写操作都同步）
- 文件会**持续增长**，一年可能达到几十 GB

**AOF 特点**：
| 优点 | 缺点 |
|------|------|
| 数据安全性高 | 文件体积大 |
| 可配置不同刷盘策略 | 恢复速度慢（要回放命令） |

**刷盘策略**：
```conf
appendfsync always    # 每条命令都同步（最安全，最慢）
appendfsync everysec  # 每秒刷盘（推荐，平衡）
appendfsync no        # 由操作系统决定（最快，最不安全）
```

---

### 问 3：AOF 重写

**问题**：对同一个 key 执行 1000 次 `INCR counter`，AOF 文件会记录 1000 条命令。如何优化？

**答案**：**AOF 重写（BGREWRITEAOF）**

```bash
# 原始 AOF 记录（1000 条命令）
INCR counter
INCR counter
...
INCR counter

# 重写后（1 条命令）
SET counter 1000
```

**重写过程**：
1. 后台运行 `BGREWRITEAOF`
2. 分析 AOF 文件，合并重复命令
3. 生成"最小命令集"的新文件
4. 替换旧文件

---

### 问 4：混合持久化（推荐）

**问题**：RDB 和 AOF 各有什么优缺点？生产环境应该如何选择？

**答案**：**两者结合**，兼顾速度和完整性。

```
┌─────────────────────────────────────────────────────────┐
│                    混合持久化                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  RDB 快照（每 5 分钟）                                   │
│  ├── 保存完整数据快照                                    │
│  └── 用于快速恢复 + 备份                                 │
│                                                         │
│  AOF 日志（每秒刷盘）                                    │
│  ├── 记录两次快照之间的写命令                            │
│  └── 用于数据完整性                                      │
│                                                         │
│  重启恢复流程：                                          │
│  1. 加载最近的 RDB 快照（快速）                          │
│  2. 回放 RDB 之后的 AOF 命令（完整）                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

---

### 问 5：COW 机制（Copy-On-Write）

**问题**：RDB 快照期间，如果有写请求，会影响快照一致性吗？

**答案**：Redis 使用 **COW（写时复制）** 机制。

```
第 5 分钟：开始 RDB 快照
├── Redis fork() 一个子进程
├── 子进程：拥有内存的"快照"（只读）
└── 主进程：继续处理写请求
    ├── 修改数据时，复制要修改的页
    └── 原数据页保持不变（供子进程读取）
```

**类比前端**：
```js
// 浅拷贝 - 类似 COW
let original = { a: 1, b: 2 }
let snapshot = { ...original }  // 快照

// 修改原对象
original.a = 999  // 不影响快照

// snapshot.a 仍然是 1
```

---

### 问 6：COW 的内存消耗

**问题**：Redis 有 10GB 数据，执行 RDB 快照期间：
- 没有写操作：COW 复制多少内存？
- 50% 数据被修改：COW 复制多少内存？

**答案**：
| 场景 | COW 额外消耗 |
|------|-------------|
| 没有写操作 | ≈ 0（父子进程共享内存） |
| 修改 10% 数据 | ≈ 1GB |
| 修改 50% 数据 | ≈ 5GB |
| 修改 100% 数据 | ≈ 10GB（最坏情况） |

**风险**：
- Redis 内存快满时执行 RDB
- COW 可能申请额外内存
- 触发系统 OOM，Redis 被杀死

**解决**：高峰期禁用 RDB 快照

---

### 问 7：生产环境配置

**场景**：电商网站，日均订单 100 万，可接受丢失最多 5 分钟数据，高峰期 10:00-12:00 和 20:00-22:00。

**推荐配置**：
```conf
# RDB 快照
save 300 10000    # 5 分钟内至少 1 万次修改才快照
save 900 1        # 15 分钟内至少 1 次修改才快照

# AOF 刷盘
appendfsync everysec  # 每秒刷盘

# 高峰期禁用 RDB（通过定时任务）
# 10:00-12:00, 20:00-22:00 执行：
# redis-cli CONFIG SET save ""
```

## 十二、Redis 实际使用场景

---

### 问 1：缓存读写策略

**问题**：电商商品详情页，用 Redis 做缓存，什么时候读/写/删除缓存？

**答案**：

**读取缓存**：
```
用户请求 → 先查 Redis → 命中则返回
          ↓ 未命中
        查数据库 → 写入缓存 → 返回
```

**写入缓存策略**：

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| **立即缓存** | 第 1 次访问就写入 | 热门数据居多 |
| **延迟缓存** | 访问 N 次后才写入 | 冷门数据多，避免缓存污染 |

**删除缓存策略**：

| 策略 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **删除缓存** | 修改 DB 后删除缓存 | 简单可靠 | 下次访问会慢 |
| **更新缓存** | 修改 DB 后更新缓存 | 下次访问快 | 并发冲突风险 |
| **自动过期** | 设置过期时间 | 无需手动删除 | 过期前数据不一致 |

**推荐**：**删除缓存 + 自动过期**组合

---

### 问 2：缓存穿透

**问题**：恶意用户访问不存在的商品 ID（缓存和数据库都没有），会发生什么？

**答案**：

**缓存穿透**：
- 查询**不存在**的数据
- 缓存里没有，数据库里也没有
- 每次请求都打到数据库
- 大量恶意请求 → 数据库崩溃

**解决方案**：
```bash
# 把"空结果"也缓存起来
SET product:999999 "NULL" EX 60   # 缓存 1 分钟
```

**其他方案**：
- 布隆过滤器（Bloom Filter）：拦截不存在的 ID
- 参数校验：ID 必须为正整数等

---

### 问 3：缓存击穿

**问题**：热门商品（iPhone 15）缓存突然过期，10000 个并发请求同时访问，会发生什么？

**答案**：

**缓存击穿**：
- **某个热点 key**突然过期
- 大量并发请求同时打到数据库

**解决方案**：

**方案 A：互斥锁**
```js
let value = redis.get(key)
if (!value) {
    if (tryLock(key + "_lock")) {
        value = db.query(key)
        redis.set(key, value)
        unlock(key + "_lock")
    } else {
        sleep(50)
        value = redis.get(key)
    }
}
```

**方案 B：永不过期**
- 设置"逻辑过期"，不设物理过期
- 后台定时任务异步更新

---

### 问 4：缓存雪崩

**问题**：Redis 宕机了，或者大量缓存同时过期，会发生什么？

**答案**：

**缓存雪崩**：
- **大量 key 同时过期**或**Redis 宕机**
- 所有请求打到数据库
- 数据库瞬间崩溃

**解决方案**：
1. **过期时间随机化**：`expire = base + random(1, 300)`
2. **Redis 集群**：多台互为备份
3. **限流降级**：数据库压力大时限制请求

---

### 问 5：计数器/排行榜

**问题**：文章点赞功能，每秒 1000 次点赞，如何优化？

**答案**：

**传统做法**：
```sql
UPDATE articles SET likes = likes + 1 WHERE id = 1001
```
- 每次都写数据库 → 大量并发写入 → 性能瓶颈

**Redis 优化**：
```bash
# Redis INCR 原子操作
INCR article:1001:likes

# 每秒同步一次到数据库
SET article:1001:likes 1000 EX 1
```

**优点**：
- `INCR` 是原子操作，无并发问题
- 内存操作，性能极高
- 定期批量写入数据库，减少 DB 压力

**数据持久化**：
- RDB + AOF 混合策略
- 宕机时最多丢失 1 秒数据

## 十三、三大缓存问题对比

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **缓存穿透** | 查询**不存在**的数据 | 缓存空结果、布隆过滤器 |
| **缓存击穿** | **某个热点 key**过期 | 互斥锁、永不过期 |
| **缓存雪崩** | **大量 key 同时过期**或 Redis 宕机 | 随机过期时间、Redis 集群 |

## 十五、Redis 集群和主从复制

---

### 问 1：如何扩展 Redis 容量？

**问题**：Redis 单机内存 16GB，QPS 5 万。业务发展后，内存需要 20GB，QPS 达到 10 万。如何扩展？

**答案**：**加机器**，有两种主流方案。

---

### 方案 A：主从复制

```
       主节点 (Master)
      /    |    \
   从 1    从 2    从 3
```

**特点**：
- 主节点负责**写**，从节点负责**读**（读写分离）
- 从节点挂了 → 不影响写入，只是读能力下降
- 主节点挂了 → **无法写入**（需要人工干预切换）

**配置**：
```conf
# 从节点配置
replicaof 192.168.1.100 6379
```

---

### 方案 B：哨兵模式（Sentinel）

**问题**：主节点挂了，如何自动恢复写入能力？

**答案**：**哨兵模式** —— 自动故障转移。

```
       哨兵 1    哨兵 2    哨兵 3
          |       |       |
          v       v       v
       主节点 → 从节点 1 → 从节点 2
```

**哨兵的作用**：
1. **监控**：不断检查主节点和从节点是否健康
2. **通知**：主节点挂了，通知管理员
3. **自动故障转移**：投票选举一个新主节点

**Quorum（多数派原则）**：
- 至少 `(N/2 + 1)` 个哨兵同意，才能判定主节点挂了
- 3 个哨兵 → 至少 2 个同意
- 5 个哨兵 → 至少 3 个同意
- **哨兵数量必须是奇数**（避免脑裂）

---

### 方案 C：分片集群（Redis Cluster）

**问题**：数据有 100GB，单机内存只有 16GB。需要多少台机器？客户端怎么知道 key 存在哪？

**答案**：需要 **7 台**（100GB / 16GB ≈ 6.25，向上取整）。

Redis 使用 **哈希槽（Hash Slot）** 来分配数据：

```
┌─────────────────────────────────────────────────────────┐
│              Redis Cluster (16384 个槽)                  │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  节点 A: 槽 0 ~ 5000                                    │
│  节点 B: 槽 5001 ~ 10000                                │
│  节点 C: 槽 10001 ~ 16383                               │
│                                                         │
│  存数据：CRC16(key) % 16384 → 定位到槽 → 定位到节点      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**客户端不需要问**，自己就能算出来：
```
key = "user:1001"
slot = CRC16("user:1001") % 16384 = 1234
→ 槽 1234 属于节点 A
→ 直接访问节点 A
```

---

### 集群扩容

**问题**：原来 3 个节点（A/B/C），现在要加 3 个节点（D/E/F）。数据怎么迁移？

**答案**：

**槽重新分配**：
```
原集群：
A: 0-5000    B: 5001-10000    C: 10001-16383

新集群：
A: 0-2500    D: 2501-5000
B: 5001-7500 E: 7501-10000
C: 10001-12500 F: 12501-16383
```

**迁移过程**：
1. 源节点锁定要迁移的槽（只读，不能写）
2. 批量迁移数据到目标节点
3. 迁移完成后，更新集群元数据
4. 客户端收到"重定向"响应，自动访问新节点

**注意**：迁移槽的瞬间，写入会短暂阻塞。

---

### MOVED 响应（客户端重定向）

**问题**：槽从节点 A 迁移到节点 D 后，客户端还在访问节点 A，会发生什么？

**答案**：类似 HTTP 301 重定向。

```bash
# 客户端请求
GET user:1001  → 访问节点 A

# 节点 A 回应
MOVED 3001 192.168.1.5:6379  # "去节点 D 找我"
```

**智能客户端的做法**：
1. 第一次收到 `MOVED` 响应 → 重定向到新节点
2. **更新本地路由表** → 记住这个槽现在属于新节点
3. 下次直接访问新节点，不再走弯路
