**生**：不了解

**师**：好的，我们来换个角度思考。假设你是仓库管理员，有 1000 个包裹，每个包裹上贴了过期时间。你会怎么做？

- 方案 A：每分钟检查所有包裹，把过期的扔掉
- 方案 B：有人来取包裹时，顺便检查这个包裹是否过期

**生**：导致可能会一直占用内存。久而久之空间被塞满

**师**：很好，你已经发现了单纯用方案 B 的问题。那么现在你来设计：如何结合方案 A 和方案 B 的优点，既不会太耗 CPU，又能保证过期数据最终被清理？

**师**：非常好，你已经自己想出了 Redis 的过期删除策略！Redis 确实是这么做的：

- **定期删除**：每隔一段时间随机抽查一部分数据，删除过期的
- **惰性删除**：访问时也会检查是否过期

这种组合策略平衡了 CPU 和内存的开销。

## 五、Redis 数据结构

### 问 1：简单值存储（String）

### 问 2：对象存储（Hash）

**师**：现在假设你要存储一个**用户对象**，包含 `name`、`email`、`age` 三个字段。

前端你会这样写：
```js
const user = {
  name: "zhangsan",
  email: "zhang@example.com",
  age: 25
}
```

在 Redis 中，你觉得用什么数据结构来存储这种**有多个字段的对象**？

提示：Redis 有一种数据结构叫 **Hash**，它和前端 JavaScript 的什么概念很像？

**生**：Hash 结构，相当于 js 的 object

---

**师**：完全正确！Hash 就类似于 JS 的 Object。那么我来问你：

在前端，你访问对象的属性会这样写：
```js
user.name  // "zhangsan"
user.email // "zhang@example.com"
```

在 Redis 中，如果要实现类似的操作，你觉得命令会是什么格式？

提示：Redis 的 Hash 命令以 `H` 开头，比如 `HSET`（设置字段）、`HGET`（获取字段）。

试着猜一下，存储上面这个 `user` 对象，命令应该怎么写？

**生**：HSET user {name: "zhangsan", email: "zhang@example.com"}

---

**师**：很好的尝试！你已经理解了 Hash 的概念，但语法需要调整一下。

Redis Hash 不是存 JSON 字符串，而是**逐个字段存储**的：

```bash
HSET user:1001 name "zhangsan"
HSET user:1001 email "zhang@example.com"
HSET user:1001 age 25
```

或者一条命令设置多个字段：
```bash
HSET user:1001 name "zhangsan" email "zhang@example.com" age 25
```

---

**师**：类比前端的 `user.name` 和 `user.email`，Redis 中获取这些字段的命令是什么？

猜一下：
- 获取单个字段（类似 `user.name`）→ `H???`
- 获取所有字段（类似 `Object.entries(user)`）→ `H???`

**生**：HGET

---

**师**：正确！

```bash
HGET user:1001 name   # 获取单个字段
HGETALL user:1001     # 获取所有字段
```

### 问 3：队列场景（List）

**师**：现在思考一个新的数据结构。

前端你有这样一个场景：
```js
// 消息队列 - 先进先出
const queue = ["msg1", "msg2", "msg3"]
queue.shift() // 从头部取出 "msg1"
queue.push("msg4") // 从尾部添加
```

Redis 中有一个数据结构叫 **List**，它支持从**头部**和**尾部**分别进行插入和弹出操作。

你觉得 List 适合实现什么功能？提示：从"先进先出"或"后进先出"的角度思考。

**生**：比较适合查询、文件上传队列？

---

**师**：很好，你想到了**队列**场景！

Redis List 确实适合：
- **消息队列**（先进先出）
- **任务队列**（如文件上传/处理队列）
- **最近浏览记录**（后进先出，类似栈）

---

**师**：继续思考 List 的操作命令。

前端操作数组：
```js
queue.push("msg")    // 从尾部添加
queue.shift()        // 从头部取出（先进先出）
queue.unshift("msg") // 从头部添加
queue.pop()          // 从尾部取出（后进先出）
```

Redis List 的命令格式是 `L` 开头（Left）和 `R` 开头（Right），分别代表**头部**和**尾部**。

猜一下：
- 从**尾部**插入（类似 `push`）→ `R???`
- 从**头部**取出（类似 `shift`）→ `L???`

**生**：RPUSH  LSHFIT

---

**师**：几乎正确！`RPUSH` 是对的，但 Redis 中"从头部取出"的命令是 `LPOP`（不是 LSHIFT）：

```bash
RPUSH queue "msg1" "msg2" "msg3"   # 从尾部插入
LPOP queue                         # 从头部取出（先进先出）
```

这样组合就实现了**队列**（FIFO）。

---

### 问 4：去重场景（Set）

**师**：现在思考一个新的场景。

前端你有这样一个需求：
```js
// 存储用户的标签，不能重复
const tags = new Set(["js", "vue", "redis"])
tags.add("js")  // 不会重复添加
```

Redis 中 **Set** 数据结构的特点是什么？它和 List 的主要区别在哪里？

**生**：特点是去重，单一性。

---

**师**：正确！Set 的核心特点是**自动去重**。

Set 还有一个强大的功能 —— **集合运算**。

前端场景：
```js
const A = new Set([1, 2, 3, 4])
const B = new Set([3, 4, 5, 6])

// 交集：[3, 4]
// 并集：[1, 2, 3, 4, 5, 6]
// 差集：A - B = [1, 2]
```

Redis Set 支持 `SINTER`（交集）、`SUNION`（并集）、`SDIFF`（差集）命令。

你觉得这种**集合运算**在实际业务中有什么应用场景？

提示：想象你有两个用户集合 —— "关注列表 A"和"关注列表 B"。

**生**：比如说抖音的共同关注，共同好友

---

**师**：非常好的例子！你理解了 Set 的实际用途。

**共同关注/共同好友** 就是典型的**交集运算**：
```bash
SINTER follow:user1 follow:user2  # 共同关注
```

其他应用场景：
- **并集**：合并两个标签系统的标签
- **差集**：找出"关注了你但你没关注的人"（互粉检测）

---

### 问 5：排行榜场景（ZSet）

**师**：最后一种数据结构 —— **ZSet**（Sorted Set，有序集合）。

前端场景：
```js
// 游戏排行榜：玩家 ID + 分数，按分数排序
const leaderboard = [
  { id: "player1", score: 100 },
  { id: "player2", score: 300 },
  { id: "player3", score: 200 },
]
// 按分数排序后：player2 > player3 > player1
```

ZSet 和 Set 的区别是：每个元素都有一个**分数（score）**，可以按分数排序。

你觉得**排行榜**这种功能，用 ZSet 实现有什么好处？提示：从"每次都要重新排序"vs"自动保持有序"的角度思考。

**生**：用 Zset 的话自动排序，不需要每次查时进行排序，直接就是有序的，提高性能

---

**师**：完全正确！这就是 ZSet 的核心优势 —— **自动保持有序**。

```bash
ZADD leaderboard 100 "player1"
ZADD leaderboard 300 "player2"
ZADD leaderboard 200 "player3"

# 直接获取前 3 名（已排序，无需额外排序）
ZREVRANGE leaderboard 0 2 WITHSCORES
```

传统数据库需要 `ORDER BY score DESC` 每次查询都排序，而 ZSet 插入时就已经排好序了。

## 七、持久化深入（RDB vs AOF）

### 问 1：RDB 快照的数据丢失问题

**师**：之前我们学过，Redis 有两种持久化方式的"雏形"：

- **方案 A**：每隔一段时间把整个内存数据打包存到硬盘
- **方案 B**：每次写操作都记录到硬盘

我们当时得出的折中方案是"缓冲区 + 每秒刷盘"。

现在我来问你：如果 Redis 采用**方案 A**（全量快照），假设每 5 分钟保存一次完整数据到硬盘。

- 第 0 分钟：Redis 启动，有 1000 条数据
- 第 2 分钟：新增了 500 条数据
- 第 4 分钟：又新增了 500 条数据
- 第 5 分钟：触发快照保存
- 第 5 分 30 秒：Redis 宕机

**问题**：重启后，Redis 会丢失多少数据？

**生**：快照保存的意思还没有完整保存到硬盘吗

---

**师**：好问题！让我澄清一下：

**第 5 分钟触发快照保存** = 在第 5 分钟这一刻，把当时 Redis 里的所有数据（1000 + 500 + 500 = 2000 条）完整保存到硬盘。

所以第 5 分 30 秒宕机时：
- 硬盘里有第 5 分钟保存的 2000 条数据

### 问 2：AOF 持久化方式

**师**：现在思考另一种方式 —— **AOF（Append Only File）**。

AOF 的思路是：每次写操作都记录到硬盘的一个文件里，类似"记账"。

```bash
# 用户执行
SET username "zhangsan"
HSET user:1001 name "lisi"
DEL old_key

# AOF 文件里记录
*2\r\n$3\r\nSET\r\n$8\r\nusername\r\n$8\r\nzhangsan\r\n
*3\r\n$4\r\nHSET\r\n$8\r\nuser:1001\r\n$4\r\nname\r\n$4\r\nlisi\r\n
*2\r\n$3\r\nDEL\r\n$7\r\nold_key\r\n
```

**AOF 的特点**：

- 文件会**持续增长**，一年可能达到几十 GB

### 问 5：COW 机制

**师**：现在思考一个实际问题。

**场景**：Redis 正在执行 RDB 快照（把内存数据写入硬盘），但此时有客户端发来写请求。

```bash
# 第 5 分钟：开始 RDB 快照（需要 10 秒）
# 第 5 分 05 秒：用户执行 SET key "new_value"
# 第 5 分 10 秒：RDB 快照完成
```

**问题**：
1. 第 5 分 05 秒的写操作，会影响正在进行的 RDB 快照吗？
2. 如果会影响，快照里的数据是不是一致的？
3. Redis 可能用什么机制来解决这个问题？

提示：类比前端的"深拷贝"和"浅拷贝"。

**生**：应该不会影响。用锁机制来解决

---

**师**：很好的直觉！用锁来保证一致性是一个思路。

但 Redis 的做法更巧妙 —— **写时复制（Copy-On-Write，简称 COW）**：

```
第 5 分钟：开始 RDB 快照
├── Redis  fork() 一个子进程
├── 子进程：拥有内存的"快照"（只读）
└── 主进程：继续处理写请求
    ├── 修改数据时，复制要修改的页
    └── 原数据页保持不变（供子进程读取）

第 5 分 10 秒：子进程完成快照，退出
```

**类比前端**：
```js
// 浅拷贝 - 类似 COW
let original = { a: 1, b: 2 }
let snapshot = { ...original }  // 快照

// 修改原对象
original.a = 999  // 不影响快照

// snapshot.a 仍然是 1
```

### 问 6：COW 的内存消耗

**师**：继续思考 COW 的问题。

**场景**：Redis 有 10GB 数据，开始 RDB 快照后：
- 如果**没有写操作**：COW 复制多少内存？
- 如果**大量写操作**（比如 50% 的数据被修改）：COW 复制多少内存？

**问题**：
1. COW 在什么情况下消耗内存最多？
2. 如果 Redis 内存本来就快满了，执行 RDB 快照会发生什么风险？

**生**：没有写操作，复制 10GB 内存？有大量写操作，15GB？

---

**师**：很好的尝试！但答案正好相反：

**COW 机制**：
- **没有写操作**：子进程和父进程**共享**同一份内存，额外消耗 ≈ 0
- **大量写操作**（50% 数据被修改）：需要复制被修改的页，额外消耗 ≈ 5GB（10GB × 50%）

```
┌─────────────────────────────────────────────────────────┐
│                  COW 内存消耗                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  fork() 瞬间：父子进程共享 10GB（不复制）                │
│                                                         │
│  写操作发生时：                                         │
│  ├── 修改 10% 数据 → 额外消耗 1GB                        │
│  ├── 修改 50% 数据 → 额外消耗 5GB                        │
│  └── 修改 100% 数据 → 额外消耗 10GB（最坏情况）          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**风险**：
- 如果 Redis 内存快满了（比如 10GB 用了 9GB）
- 执行 RDB 快照期间有大量写操作
- COW 可能申请额外 5GB+ 内存
- **结果**：触发系统 OOM（内存溢出），Redis 被杀死

```js
let value = redis.get(key)

        value = db.query(key)
        redis.set(key, value)
        unlock(key + "_lock")
    } else {
        sleep(50)  // 等其他线程查完
        value = redis.get(key)
    }
}
```

**方案 B：永不过期**

### 问 2：主从复制

**师**：先思考**主从复制**。

**场景**：1 个主节点（写）+ 3 个从节点（读）。

```bash
# 客户端读写分离
SET key "value"      # 写入主节点
GET key              # 从从节点读取
```

**问题**：
1. 主节点的数据如何同步到从节点？
2. 如果从节点挂了，会影响写入吗？
3. 如果主节点挂了，会发生什么？

**生**：主节点数据复制到从节点。从节点挂了不影响写入。主节点挂了，就会出现写入不了

---

**师**：完全正确！你已经理解了主从复制的核心特点：

**主从复制特点**：
- 主节点负责**写**，从节点负责**读**
- 从节点挂了 → 不影响写入，只是读能力下降
- 主节点挂了 → **无法写入**（需要人工干预切换）

### 问 3：哨兵模式（自动故障转移）

**师**：主节点挂了无法写入，这是个问题。

**思考**：如果主节点挂了，如何让系统自动恢复写入能力？

提示：3 个从节点中，能不能选一个"新主"？谁来选？怎么选？

**生**：如果挂了，应该是能自动恢复的。选一个性能最好的？

---

**师**：很好的思路！选一个从节点提升为主节点。

但问题是：**谁来选举**？怎么保证大家选的是一致的？

**Redis 哨兵模式（Sentinel）**：

```
       哨兵 1    哨兵 2    哨兵 3
          |       |       |
          v       v       v
       主节点 → 从节点 1 → 从节点 2
```

**哨兵的作用**：
1. **监控**：不断检查主节点和从节点是否健康
2. **通知**：主节点挂了，通知管理员
3. **自动故障转移**：投票选举一个新主节点

---

### 问 4：哨兵选举机制

**师**：思考哨兵选举的细节。

**场景**：3 个哨兵监控 1 个主节点。

```bash
# 主节点挂了
# 哨兵 1 认为主节点挂了
# 哨兵 2 认为主节点挂了
# 哨兵 3 网络慢，还没反应过来
```

**问题**：
1. 至少要几个哨兵认为主节点挂了，才能触发选举？
2. 如果有 2 个哨兵，都认为自己应该选主，怎么避免"脑裂"？

提示：类似分布式系统的"多数派"原则。

**生**：至少两个？让第三个来投票

---

**师**：完全正确！这就是**多数派原则**（Quorum）。

**哨兵选举规则**：
- 至少 `(N/2 + 1)` 个哨兵同意，才能判定主节点挂了
- 3 个哨兵 → 至少 2 个同意
- 5 个哨兵 → 至少 3 个同意

**避免脑裂**：
- 哨兵数量必须是**奇数**（1、3、5...）
- 确保任何时候都能选出"多数派"

### 问 5：分片集群（Redis Cluster）

**师**：主从复制 + 哨兵，解决了**高可用**问题，但没解决**容量**问题。

**场景**：数据有 100GB，单机内存只有 16GB。

```
主节点 (16GB) + 从节点 (16GB)
→ 最多存 16GB 数据（从节点是副本）
```

**问题**：
1. 100GB 数据，需要多少台 16GB 的 Redis？
2. 客户端怎么知道某个 key 应该存到哪个节点？

提示：类似前端的"分治"思想 —— 把数据拆分到多台机器。

**生**：7 台。客户端询问某个节点

---

**师**：很好，你算出了需要 7 台（100GB / 16GB ≈ 6.25，向上取整 7 台）。

但"客户端询问某个节点"有个问题：如果客户端要先问"这个 key 存在哪"，那每次操作都要多一次网络请求，太慢了。

**Redis Cluster 分片集群**：

Redis 的做法更聪明 —— **哈希槽**（Hash Slot）。

```
┌─────────────────────────────────────────────────────────┐
│              Redis Cluster (16384 个槽)                  │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  节点 A: 槽 0 ~ 5000                                    │
│  节点 B: 槽 5001 ~ 10000                                │
│  节点 C: 槽 10001 ~ 16383                               │
│                                                         │
│  存数据：CRC16(key) % 16384 → 定位到槽 → 定位到节点      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**客户端不需要问**，自己就能算出来：
```
key = "user:1001"
slot = CRC16("user:1001") % 16384 = 1234
→ 槽 1234 属于节点 A
→ 直接访问节点 A
```

### 问 6：集群扩容

**师**：思考**集群扩容**。

**场景**：原来 3 个节点（A/B/C），现在要加 3 个节点（D/E/F）。

```
原集群：
A: 0-5000    B: 5001-10000    C: 10001-16383

新集群：
A: 0-2500    D: 2501-5000
B: 5001-7500 E: 7501-10000
C: 10001-12500 F: 12501-16383
```

**问题**：
1. 槽的分配变了，槽里的**数据**怎么从旧节点迁移到新节点？
2. 迁移过程中，客户端还能正常读写吗？

提示：类似"搬仓库"—— 一边营业一边搬迁。

**生**：A C E  B  D  F。将 A 的后半部分迁移到 B，C 的后半部分迁移到 D，E 的后半部分迁移到 F。能正常读写

---

**师**：你的思路是对的 —— 从每个旧节点迁移一部分槽到新节点。

不过你说的 "A 的后半部分迁移到 B" 有点小误会，应该是：
- A 的后半部分 → 迁移到 **D**（新节点）
- B 的后半部分 → 迁移到 **E**（新节点）
- C 的后半部分 → 迁移到 **F**（新节点）

**关于"能否正常读写"**：

你说"能正常读写"，基本正确！但有一个细节：

**Redis 集群迁移过程**：
1. 源节点锁定要迁移的槽（只读，不能写）
2. 批量迁移数据到目标节点
3. 迁移完成后，更新集群元数据
4. 客户端收到"重定向"响应，自动访问新节点

**所以**：
- 大部分时间可以正常读写 ✅
- 但**迁移槽的瞬间**，写入会短暂阻塞 ⚠️

### 问 7：客户端如何感知集群变化

**师**：思考**客户端如何感知集群变化**。

**场景**：槽从节点 A 迁移到节点 D 后，客户端还在访问节点 A。

```bash
# 客户端请求
GET user:1001  → 访问节点 A

# 节点 A 回应
MOVED 3001 192.168.1.5:6379  # "去节点 D 找我"
```

**问题**：
1. 客户端收到 `MOVED` 响应后，应该怎么做？
2. 智能的 Redis 客户端会怎么处理？

提示：类似 HTTP 的 301 重定向。

**生**：收到响应后定向去 D 查找

---

**师**：正确！就是类似 HTTP 301 重定向。

**智能客户端的做法**：
1. 第一次收到 `MOVED` 响应 → 重定向到新节点
2. **更新本地路由表** → 记住这个槽现在属于新节点
3. 下次直接访问新节点，不再走弯路
